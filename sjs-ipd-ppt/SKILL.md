# IPD 项目评审 PPT 生成

为数据技术所生成基于 IPD 流程的项目评审汇报 PPT。覆盖 Charter（立项）、PDCP（总体设计方案评审）、ADCP（可获得性评审）、转移四个阶段。

**核心理念：** 大部分研发人员知道自己在做什么，但很难讲清楚"为什么要做"和"价值在哪"。这个 Skill 的核心价值就是帮助用户把技术工作翻译成有说服力的评审语言——特别是必要性、重要性和投入产出。

---

## 配置（首次使用必读）

首次使用时，需要确认以下配置。检查 Skill 目录下是否存在 `config.json`，如不存在则引导用户完成配置。

### 配置项

| 配置项 | 说明 | 示例 |
|--------|------|------|
| `template_path` | 公司统一 PPT 模板路径 | `/path/to/company-template.pptx` |
| `project_base_dir` | 项目资料存放根目录 | `/path/to/项目/` |
| `cost_per_day` | 人力成本基准（元/人天） | `970` |
| `product_lines` | 产品线及对应模板映射 | 见下方示例 |

### 配置引导流程

当 `config.json` 不存在时，主动询问用户：

1. **PPT 模板**："请问公司统一 PPT 模板文件在哪个路径？（如 .pptx 文件）"
2. **项目资料目录**："项目资料统一存放在哪个目录下？（用于确定输出路径）"
3. **人力成本**："人力成本基准是多少元/人天？（默认 970）"
4. **产品线**："有哪些产品线？各产品线有独立模板吗？"

将结果保存为 `config.json`（保存在 Skill 目录下）：

```json
{
  "template_path": "sjs-ipd-ppt/assets/template.pptx",
  "project_base_dir": "/path/to/projects/",
  "cost_per_day": 970,
  "product_lines": {
    "ml": {
      "template": "/path/to/ml-template.pptx"
    },
    "ac": {
      "template": "/path/to/ac-template.pptx"
    }
  }
}
```

如用户无需生成 PPT（只输出结构化要点），可跳过模板配置。

### 输出路径引导

生成 PPT 时，按以下优先级确定输出路径：

1. **用户明确指定** → 直接使用
2. **引导询问**："PPT 保存到哪个目录？建议放在项目资料目录下，例如：`{project_base_dir}/{项目名}/`"
3. **默认** → 当前工作目录

## 触发与项目识别

用户触发后，先确认以下信息：

**必填信息：**
- 项目名称（如"冰箱凝霜程度识别技术研究"）
- 产品线归属（ml / ac）→ 决定使用哪套模板
- 项目类别（基础研究 / 应用研究）→ 决定内容侧重点
- 评审阶段（Charter / PDCP / ADCP / 转移）

**如用户未提供，主动询问：**
- "请提供项目名称"
- "这个项目是 ml 产品线还是 ac 产品线的？"
- "项目类型是基础研究还是应用研究？"
- "需要准备哪个阶段的材料？（Charter/PDCP/ADCP/转移）"

## 工作模式：引导式为主

**默认采用引导式**——用户往往只能说出"做了什么"，需要协助提炼"为什么做"和"价值是什么"。

引导流程：
1. 先让用户自由描述项目（不限格式，怎么说都行）
2. 从描述中提取已知信息
3. 针对缺失的"价值论证"部分，用提问引导用户思考
4. 将用户的回答加工成有说服力的评审语言
5. 逐章节确认，再进入下一章

**填充式（仅在用户主动提供完整数据时）：** 用户直接给出结构化数据（指标、费用、进度），跳过引导直接生成。

## 项目类型差异

| 维度 | 基础研究 | 应用研究 |
|------|---------|---------|
| 核心驱动 | 技术前瞻、行业技术布局 | 业务/产品明确需求 |
| 立项价值论证 | 技术趋势、能力储备、降本提效空间 | 业务痛点、用户价值、投入产出 |
| 成果形态 | 算法/模型/技术平台/行业标准 | 系统应用/产品功能/业务工具 |
| 推广方式 | 技术复用到多产品线/多场景 | 直接解决具体业务或产品问题 |

**价值引导差异：**
- 基础研究 → 引导用户思考"技术趋势是什么？不做的后果？做了能获得什么能力？降本提效空间多大？"
- 应用研究 → 引导用户思考"谁需要这个？解决什么痛点？业务指标怎么改善？投入产出比多少？"

## 各阶段详细指南

> 详细指南见 `references/` 目录下的各阶段文档。以下为速查概要。

### 1. Charter（立项）— 核心是讲清"为什么要做"

Charter 材料的灵魂是**必要性、重要性、投入产出**。技术细节可以简略，但价值论证必须充分。

**结构（8大模块）：**

| 序号 | 模块 | 占比 | 内容要点 |
|------|------|------|---------|
| 1 | 项目背景 | 15% | 行业趋势、技术现状、痛点问题、需求来源 |
| 2 | 研究内容 | 15% | 技术路线、研究方法概述（不需要太细） |
| 3 | **项目价值** | **25%** | **能力提升、经济效益、应用范围、技术突破（重点！）** |
| 4 | 项目目标 | 15% | Q/C/D 指标 |
| 5 | 关键资源计划 | 5% | 人力、设备概要 |
| 6 | 风险及问题管理 | 5% | 关键风险 |
| 7 | 财务计划 | 10% | 费用概算 |
| 8 | 结论 | 10% | 总结必要性、下一步行动 |

**引导式提问（逐层深入）：**

第一层（了解项目）→ "简单描述一下这个项目是做什么的？"

第二层（挖掘价值）→ 针对基础研究："行业做到什么程度？不做的后果？能支撑哪些应用？"
→ 针对应用研究："谁的需求？现在怎么做的？痛点？指标能改善多少？"

第三层（补充数据）→ Q 指标、人天（×cost_per_day=人力成本）、其他费用、里程碑

**价值提炼框架：** 技术价值 + 经济价值 + 应用价值 + 战略价值

> 完整指南：`references/charter-guide.md`

### 2. PDCP（总体设计方案评审）— 核心是讲清"怎么做"

**结构（9大模块）：** 项目简介 → 技术目标 → **研究方法** → **技术方案** → 资源 → 风险 → 知识产权 → 环境影响 → 成员

**重点：** 技术方案和研究方法占 55%，需要协助用户组织架构图、流程图、对比表。

> 完整指南：`references/pdcp-guide.md`

### 3. ADCP（可获得性评审/结项）— 核心是讲清"做到了什么"

**结构（8大模块）：** 项目简介 → **Q/C/D 偏差分析** → 转移计划 → 资源 → 转移风险 → 技术创新 → 经济效益 → 过程运作

**重点：** 偏差分析占 35%，逐项对比"计划 vs 实际"。

> 完整指南：`references/adcp-guide.md`

### 4. 转移阶段 — 核心是讲清"怎么落地"

**结构（5大模块）：** 项目简介 → 转移偏差分析 → 技术成果 → 推广应用 → 过程运作

**数据所转移重点：** 系统/功能已部署到生产环境、业务指标已验证、技术成果可复用。

> 完整指南：`references/transfer-guide.md`

## PPT 生成规范

### 模板

使用 `config.json` 中配置的 `template_path` 作为基础模板。

产品线模板选择：从 `config.json` 的 `product_lines` 中匹配。

### PPT 页面规范

每页 PPT 包含：
- **标题**：章节/要点标题
- **内容区域**：结构化要点（3-5 个要点，每点 1-2 行）
- **备注区**：讲解文案（详细说明、数据支撑、过渡语）

### 内容深度

**结构化要点模式（默认）：**
- 每页 3-5 个核心要点
- 要点下可有 1-2 个子要点
- 关键数据用数字突出
- 备注中写详细讲解文案

**完整内容模式（信息充足时）：**
- 当用户提供完整数据时，使用 ppt-master 等工具生成完整 PPT
- 信息完整性判断：Q/C/D 指标、费用明细、进度数据、技术方案均已有

### 生成方式

优先使用 `scripts/gen_ppt.py` 生成：

```bash
python scripts/gen_ppt.py \
  --stage charter \
  --project "项目名" \
  --type basic \
  --template /path/to/template.pptx \
  --output /path/to/output/
```

或按以下流程手动使用 python-pptx：
1. 加载 config 中指定的模板
2. 按阶段结构添加幻灯片
3. 每页填充标题和要点
4. 在备注中添加讲解文案
5. 保存到用户确认的输出目录

## 工作流程

```
用户触发
  ↓
检查 config.json 是否存在
  ├─ 不存在 → 引导配置（模板路径、资料目录、人力成本、产品线）
  └─ 已存在 → 跳过
  ↓
确认项目信息（名称、产品线、类型、阶段）
  ↓
引导确认输出路径（建议 {project_base_dir}/{项目名}/）
  ↓
用户自由描述项目（提取已知信息）
  ↓
逐章节引导提问（重点挖掘价值论证）
  ↓
整理结构化要点 + 备注讲解文案
  ↓
确认内容 → 生成 PPT
  ↓
输出文件路径，提示用户查看
```

## 费用计算

- 人力成本基准：从 `config.json` 读取（默认 970 元/人天）
- 自动计算：人天数 × cost_per_day = 人力成本
- 其他费用需用户提供：服务器、差旅、试制、模具、专利等

## 注意事项

- 数据所项目核心关注：**投入产出、降本提效、业务/产品需求、行业技术布局**
- B级及以上项目需填写知识产权方案
- 项目编号、项目级别等信息需向用户确认
- 考核评分标准参考 `references/` 目录下的考核办法文档
