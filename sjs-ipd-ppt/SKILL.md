# IPD 项目评审 PPT 生成

为数据技术所生成基于 IPD 流程的项目评审汇报 PPT。覆盖 Charter（立项）、PDCP（总体设计方案评审）、ADCP（可获得性评审）、转移四个阶段。

**核心理念：** 大部分研发人员知道自己在做什么，但很难讲清楚"为什么要做"和"价值在哪"。这个 Skill 的核心价值就是帮助用户把技术工作翻译成有说服力的评审语言——特别是必要性、重要性和投入产出。

---

## 产品线分类（模板自动选择）

模板按阶段分为 4 个通用文件，无需用户配置。脚本根据产品线自动替换公司名和承担单位：

| 模板文件 | 适用阶段 |
|---------|---------|
| `charter-template.pptx` | Charter（立项） |
| `pdcp-template.pptx` | PDCP（设计评审） |
| `adcp-template.pptx` | ADCP（结项评审） |
| `transfer-template.pptx` | 转移 |

**产品线决定公司名和承担单位（脚本自动替换）：**

| 产品线 | 公司名 | 承担单位 | 适用范围 |
|--------|--------|---------|---------|
| **hm** | 四川虹美智能科技有限公司 | 数据技术所 | 空调、公司通用平台技术 |
| **ml** | 长虹美菱股份有限公司 | AI数智研发部 | 冰箱、洗衣机、小电日电、数字化 |

**分类判断规则（自动判断，不确定才问）：**
- 项目名称或描述中含"空调"、"通用平台" → **hm**
- 项目名称或描述中含"冰箱"、"洗衣机"、"小电"、"日电"、"美菱"、"数字化" → **ml**
- **只有无法从项目信息中判断时**，才询问用户

## 配置（首次使用必读）

首次使用时，检查 Skill 目录下是否存在 `config.json`，如不存在则引导用户完成配置。

### 配置项

| 配置项 | 说明 | 示例 |
|--------|------|------|
| `project_base_dir` | 项目资料存放根目录 | `/path/to/项目/` |
| `cost_per_day` | 人力成本基准（元/人天） | `970` |

### 配置引导流程

**首次使用：** 当 `config.json` 不存在时，询问用户：

1. **项目资料目录**："项目资料统一存放在哪个目录下？（用于确定输出路径）"
2. **人力成本**："人力成本基准是多少元/人天？（默认 970）"

将结果保存为 `config.json`（保存在 Skill 目录下）：

```json
{
  "project_base_dir": "/path/to/projects/",
  "cost_per_day": 970
}
```

**后续使用：** `config.json` 已存在时，**不问，直接用**。只在生成 PPT 前简单确认：

```
项目目录：{project_base_dir}/{项目名}/
人力成本：{cost_per_day} 元/人天
确认无误？（直接回车确认，或告诉我修改哪项）
```

用户回车或说"确认" → 跳过，直接进入工作。
用户说"改一下目录" → 更新 config.json，继续。

### 项目目录结构

每个项目在 `project_base_dir` 下有独立目录，所有产出集中存放：

```
{project_base_dir}/{项目名}/
├── 大纲/                          # 确认版大纲（带版本）
│   ├── {项目名}-CHARTER-v1.md     # 初版
│   ├── {项目名}-CHARTER-v2.md     # 迭代版
│   └── {项目名}-CHARTER-v3.md     # 最终确认版
├── PPT/                           # 生成的 PPT（带版本，与大纲版本对应）
│   ├── {项目名}-CHARTER-v1-汇报材料.pptx
│   ├── {项目名}-CHARTER-v2-汇报材料.pptx
│   └── {项目名}-CHARTER-v3-汇报材料.pptx
├── 资料/                          # 用户提供的参考资料
│   ├── 架构图.png
│   ├── 数据表.xlsx
│   └── 竞品分析.docx
└── README.md                      # 项目说明（可选）
```

**目录创建时机：** 用户提供项目名称后，立即创建项目目录及子目录。

**资料存放引导：**
- 用户提供图片/文档/表格时，引导保存到 `资料/` 目录
- 大纲引用资料路径时，使用相对路径（如 `资料/架构图.png`）
- 用户也可直接将文件拷贝到 `资料/` 目录下

## 触发与项目识别

用户触发后，先确认以下信息：

**必填信息：**
- 项目名称（如"冰箱凝霜程度识别技术研究"）
- 产品线归属（hm / ml）→ **从项目名称自动判断**，不确定时才问
- 项目类别（基础研究 / 应用研究）→ 从描述中判断
- 评审阶段（Charter / PDCP / ADCP / 转移）→ 从用户输入中提取

**自动判断逻辑：**
- 用户说"空调湿度传感器" → hm，直接确认，不问
- 用户说"冰箱凝霜识别" → ml，直接确认，不问
- 用户说"智能家电感知平台" → 不确定，询问

**只在无法判断时询问：**
- "这个项目是面向空调/通用平台（hm）还是美菱产品线（ml）的？"

## 工作模式：引导式为主

**核心目标：** 让评审一看就觉得——这个项目**必须做**、**很重要**、**很有价值**。

**工作方式：用户说，我们提炼。** 用户只需要用自己的话描述项目，我们负责加工成有说服力的评审语言。

### 引导入口（只需问三个问题）

1. **"这个项目要解决什么问题？"** — 痛点、背景、需求来源
2. **"解决了能带来什么？"** — 收益、价值、能力提升
3. **"有没有现成的资料？"** — 架构图、数据表、竞品分析等

用户哪怕只说一句话（"冰箱结霜太费电了，想做个识别算法省电"），我们就能开始提炼。

### 加工流程

1. 用户自由描述（怎么说都行，一句话也行）
2. 我们从描述中提取已知信息
3. **主动加工提炼**——把口语变成评审语言，补充量化数据
4. 针对缺失的部分，**定向追问**（不是漫无目的问，而是"这里缺一个数字，能补吗？"）
5. 逐章节确认，再进入下一章

**填充式（仅在用户主动提供完整数据时）：** 用户直接给出结构化数据（指标、费用、进度），跳过引导直接生成。

### 技术资料引导与内容完善

**不只是收集资料，更要协助用户完善内容。** 用户提供原始素材后，需要加工成有说服力的评审语言。

在了解项目概况后，主动询问用户是否有以下资料可以提供：

| 资料类型 | 说明 | 对应章节 | 加工方向 |
|---------|------|---------|---------|
| 技术架构图 | 系统/算法整体架构 | 研究内容、技术方案 | 提炼技术路线的创新点和可行性 |
| 技术路线图 | 研发阶段划分、关键节点 | 研究内容 | 补充里程碑和交付物 |
| 流程图 | 算法流程、数据处理流程 | 技术方案 | 突出技术难点和解决方案 |
| 对比分析表 | 竞品对比、方案对比 | 项目背景、技术方案 | 强化差异化优势和必要性 |
| 指标数据表 | Q/C/D 指标明细 | 项目目标 | 量化目标，对标行业水平 |
| 费用明细表 | 人力/设备/其他费用 | 财务计划 | 核算投入产出比 |
| 甘特图/里程碑 | 进度计划 | 项目目标（D指标） | 明确阶段交付 |
| 项目组织架构 | 团队成员及分工 | 资源计划 | 展示团队能力 |

引导话术（简洁版）：
- "有没有现成的资料？架构图、数据表、竞品分析什么的，直接发给我或拷贝到项目 `资料/` 目录下。"
- "有 Q/C/D 指标的表格吗？有的话直接发过来，没有的话我帮你梳理。"

**核心原则：不要一次问太多，先拿到核心信息（问题+价值），资料可以后面慢慢补。**

**用户提供资料后的处理（收集 + 完善）：**
- 图片/截图 → 保存到 `资料/` 目录，提取关键信息写入大纲，并补充评审视角的解读
- 文档/表格 → 保存到 `资料/` 目录，提炼核心数据，按 PPT 页面结构化
- 口述描述 → 加工成规范的评审语言，补充量化数据和价值论证
- 用户直接粘贴内容 → 保存到 `资料/` 目录备份，同时加工润色写入大纲

**每份资料处理后，主动追问：**
- "从这个数据看，我们的优势是 XX，要不要在价值部分重点强调？"
- "这个费用明细还缺 XX 项，要不要补上？"
- "竞品对比显示我们在 XX 方面领先，建议在背景部分突出这一点。"

## 项目类型差异

| 维度 | 基础研究 | 应用研究 |
|------|---------|---------|
| 核心驱动 | 技术前瞻、行业技术布局 | 业务/产品明确需求 |
| 立项价值论证 | 技术趋势、能力储备、降本提效空间 | 业务痛点、用户价值、投入产出 |
| 成果形态 | 算法/模型/技术平台/行业标准 | 系统应用/产品功能/业务工具 |
| 推广方式 | 技术复用到多产品线/多场景 | 直接解决具体业务或产品问题 |

**价值引导差异：**
- 基础研究 → 引导用户思考"技术趋势是什么？不做的后果？做了能获得什么能力？降本提效空间多大？"
- 应用研究 → 引导用户思考"谁需要这个？解决什么痛点？业务指标怎么改善？投入产出比多少？"

## 内容完善与价值提炼

**核心原则：** 不是简单收集用户说的话，而是协助用户把技术工作翻译成评审语言。用户说"做了个算法"，你要帮他提炼出"解决了什么问题、带来了什么价值、为什么必须现在做"。

### 一、竞品与行业对标

用户往往对竞品了解有限，需要主动协助补全。**特别是基础研究项目，技术价值的核心支撑就是行业对标——我们在哪？行业在哪？差距/领先多少？**

**引导提问：**
- "行业内有谁在做类似的事情？做到什么程度了？"
- "我们的方案跟现有方案比，优势在哪？"
- "有没有行业报告或技术白皮书可以参考？"
- "这个技术在国内/国际处于什么水平？"
- "有没有填补行业空白的点？"

**加工方式：**
- 用户说"海尔也在做" → 整理为对比表格：方案/精度/成本/成熟度，突出我们的差异化
- 用户说"行业还没有好的方案" → 强调"填补行业空白"、"技术首创"、"抢占制高点"
- 用户说"效果一般" → 转化为"现有方案的局限性"，引出本项目的必要性
- 用户说"国外有类似技术" → 强调"国产替代"、"自主可控"、"打破技术垄断"

**技术价值提炼话术：**
```
对标行业：
  "目前行业内主流方案为 XX，准确率约 70%-80%，存在 XX 局限性"
  "国内尚无成熟的 XX 方案，本项目将填补这一技术空白"

行业首创/领先：
  "本项目采用 XX 技术路线，为行业内首个 XX 方案"
  "相比行业现有方案，精度提升 XX%，达到国际先进水平"

能力建立：
  "通过本项目，建立 XX 核心技术能力，形成自主知识产权"
  "突破 XX 技术瓶颈，为后续 XX 产品线提供技术支撑"
```

**输出示例：**
```
现状：行业内主流方案为定时化霜（能耗高）和温度阈值化霜（精度不足）
差距：现有方案准确率约 70%-80%，无法满足节能需求
空白：国内尚无多传感器融合的凝霜识别方案
突破：本项目将准确率提升至 90%+，填补国内技术空白
能力：建立凝霜识别核心技术平台，可复用到洗衣机、空调等产品线
```

### 二、必要性与重要性论证

**必要性 = 不做会怎样**
- "如果不做这个项目，会有什么后果？"
- "用户投诉/能耗浪费/竞争力下降的量化数据有吗？"
- "有没有政策法规或行业标准的要求？"

**重要性 = 做了能怎样**
- "这个技术突破后，能支撑哪些产品或业务？"
- "对公司技术布局有什么战略意义？"
- "能不能形成可复用的技术平台？"

**加工示例：**
```
用户原话："冰箱结霜问题挺烦的，用户老投诉"
→ 必要性：结霜导致能效下降 15%-20%，用户投诉占比 30%，直接影响产品竞争力
→ 重要性：凝霜识别是智能家电感知能力的核心环节，可复用到洗衣机、空调等 3+ 产品线
```

### 三、投入产出与价值核算

**这是评审最关注的部分，必须有量化数据。**

**引导用户提供/确认：**
- 人力投入：多少人 × 多少天 × 单价 = 人力成本
- 其他费用：服务器、试制、差旅、专利等
- 预期收益：降本（多少钱）、提效（节省多少人/时间）、增收（增加多少销售）

**价值核算框架：**

```
投入：
  人力成本 = 人天数 × cost_per_day（默认 970 元/人天）
  其他费用 = 服务器 + 试制 + 差旅 + 专利 + ...
  总投入 = 人力成本 + 其他费用

产出（量化）：
  直接经济效益 = 年节省电费/材料费/人工费 + 年增收
  间接效益 = 用户满意度提升、故障率下降、能效等级提升

投入产出比 = 直接经济效益 / 总投入
回本周期 = 总投入 / 年节省金额
```

**如果用户说"算不清楚"：**
- 帮他估算："假设产品年销量 100 万台，每台节省 5 元电费，年节省 500 万"
- 用保守估计："按最低估计，每年至少节省 XX 万"
- 强调非经济价值："即使不算经济账，技术能力的积累和专利布局也是重要产出"

**加工示例：**
```
用户原话："应该能省点电，具体多少没算过"
→ 量化估算：按 100 万台年销量，每台化霜能耗降低 30%（约 5 度电/年），
  电价 0.6 元/度，年节省电费 = 100万 × 5 × 0.6 = 300 万元
→ 投入产出比 = 300万 / 13.62万 ≈ 22倍
```

### 四、整体说服逻辑

每个章节都要服务于"必须做、很重要、有价值"这个核心论点：

| 章节 | 说服目标 | 关键问题 |
|------|---------|---------|
| 项目背景 | **必须做** — 行业趋势 + 痛点紧迫 | 不做会怎样？ |
| 研究内容 | **能做成** — 技术路线可行 | 怎么做？ |
| 项目价值 | **很有价值** — 量化收益 + 战略意义 | 值得做吗？ |
| 项目目标 | **可衡量** — Q/C/D 指标明确 | 做到什么程度？ |
| 风险管理 | **可控** — 风险有预案 | 能管住吗？ |
| 结论 | **恳请批准** — 总结必要性 + 等级建议 | 请批准 |

**价值提炼四维度：**

1. **技术价值**（基础研究核心）：
   - 行业对标：我们在哪？行业在哪？是填补空白还是行业首创？
   - 技术突破：突破了什么瓶颈？解决了什么行业难题？
   - 能力建立：建立了什么核心技术能力？有没有不可替代性？
   - 技术壁垒：申请几项专利？能不能形成技术护城河？

2. **经济价值**（应用研究核心）：
   - 降本：年节省多少？按产品量 × 单台节省金额算
   - 增收：能溢价多少？能带来多少新销售？
   - 提效：节省多少人力/时间？换算为金额
   - 投入产出比：总投入 vs 年收益，数字越大越好

3. **应用价值**：
   - 能用在哪？单产品还是多产品线？
   - 覆盖范围：年销多少万台？潜在市场多大？
   - 可复用性：能不能平台化？能移植到哪些产品？

4. **战略价值**（基础研究的"不可不做"逻辑）：
   - **技术底座**：本项目是哪个产品/大技术规划的重要基础？不做会卡住什么？
   - 技术布局：是不是公司技术战略的关键一环？
   - 行业话语权：能不能参与标准制定？有没有行业影响力？
   - 团队能力：培养了什么技术团队？积累了多少经验？

**技术底座是基础研究最强的说服点之一——不是"做了锦上添花"，而是"不做就卡脖子"。**

**引导提问：**
- "这个技术是哪个产品线的必要基础？不做的话产品方案怎么走？"
- "公司有没有更大的技术规划？本项目在其中扮演什么角色？"
- "如果这个技术不突破，后续的 XX 项目/产品能不能落地？"
- "这个能力是不是多条产品线都需要的？不做的话每个产品线都要重复投入？"

**加工话术：**
```
作为技术底座：
  "本项目是 XX 产品线智能化升级的必要基础，不突破则产品方案无法推进"
  "作为公司 XX 技术规划的核心环节，本项目为后续 3 个产品线提供底层能力"
  "该技术是 XX 大项目的前置依赖，延迟将影响整体技术规划进度"

作为能力储备：
  "通过本项目建立 XX 核心能力，为后续 XX、XX 产品线提供技术支撑"
  "填补公司在 XX 领域的技术空白，避免未来被卡脖子"
```

### 1. Charter（立项）— 讲清"为什么要做"

**核心定位：这是项目的"生死判"——过了才存在，不过就没了。**

**评审关注：这个项目值不值得投？不做会怎样？做了能带来什么？**

**产出和收益是 Charter 的决胜点，必须亮眼、有冲击力。**

| Slide | 页面 | 重点内容 | 注意事项 |
|-------|------|---------|---------|
| 0 | 封面 | 项目名称、类别 | — |
| 1 | 目录 | 固定 | 不动 |
| **2-5** | **项目背景** | **行业趋势、痛点问题、竞品差距** | **量化紧迫性：不做会怎样？** |
| **6-8** | **研究内容** | **技术路线概述（简略）** | **点到为止，不展开技术细节** |
| **9** | **项目价值** | **技术价值、经济价值、应用价值、战略价值** | **四维度必须有量化数据** |
| 10 | Q 指标 | 指标要求/交付物 | 表格填入 |
| 11 | C 指标 | 费用科目/预算 | 表格填入 |
| 12 | D 指标 | 里程碑/交付物 | 表格填入 |
| 13 | 关键资源计划 | 人员/分工/工作量 | 表格填入 |
| 14 | 财务计划 | 费用/各阶段分配 | 表格填入 |
| **15** | **⭐ 产出和收益** | **这是评审最后看的、也是印象最深的一页** | **必须亮眼！** |
| 16 | 风险管理 | 风险/应对 | 表格填入 |
| 17 | 结论-评分 | 评价指标打分 | 表格填入 |
| 18 | 结论-总结 | **恳请IPMT批准（固定写法）** | 保留固定写法，只改总结内容 |

#### 产出和收益页——Charter 的决胜点

这一页是评审做决策的关键依据，必须让人一看就觉得"这个项目必须投"。

**必须包含的亮点（至少 3 项）：**

| 亮点类型 | 怎么写才有冲击力 | 示例 |
|---------|----------------|------|
| **降本** | 给具体金额 + 年度 + 产品范围 | "年节省电费 300 万元（覆盖 100 万台冰箱）" |
| **提效** | 量化效率提升 + 换算为人力/时间 | "化霜效率提升 30%，等效节省 5 人/年" |
| **增收** | 关联产品卖点 + 市场预期 | "新增智能节能卖点，预计溢价 2%，年增收 500 万" |
| **技术能力** | 强调平台化 + 可复用 | "核心算法可复用到洗衣机、空调，覆盖 3+ 产品线" |
| **专利** | 给数量 + 类型 | "产出发明专利 2 项，构建技术壁垒" |
| **行业地位** | 强调首创/领先 | "行业内首个 XX 方案，抢占技术制高点" |

**收益写法对比：**

```
❌ 不够亮眼："预计能降低能耗，节省一些成本"
✅ 亮眼写法："化霜能耗降低 30%，按年销 100 万台、每台节省 5 度电计算，
             年节省电费 300 万元，投入产出比 22:1"

❌ 不够亮眼："可以复用到其他产品"
✅ 亮眼写法："核心算法平台化，可直接移植到洗衣机（年销 200 万台）
             和空调（年销 150 万台），潜在年收益超 1000 万元"
```

**引导用户产出亮点的提问：**
- "降本的具体数字能算出来吗？按产品年销量 × 单台节省金额"
- "这个技术能不能形成卖点？能溢价多少？"
- "专利有几项？是发明还是实用新型？"
- "行业内有没有类似方案？如果没有，我们就是首创"
- "这个能力能不能平台化？能覆盖几个产品线？"

#### 引导重点（整体）

1. 行业痛点有多严重？（量化：投诉占比、能耗损失、效率差距）
2. 竞品做到什么程度？我们差距/优势在哪？
3. 不做这个项目会怎样？（后果量化）
4. 做了能带来什么？（**必须具体数字，不能模糊**）
5. 投入产出比多少？（总投入 vs 年收益，**数字越大越好**）
6. 项目等级预期？（A+/A/B，对应需要强化的方向）

> 完整指南：`references/charter-guide.md`

## PPT 生成规范

### 模板

模板按阶段分为 4 个通用文件（`{stage}-template.pptx`），脚本根据产品线自动替换公司名和承担单位，无需用户指定。

**四个阶段模板结构类似，均遵循以下页面规则。**

### 模板页面规则（逐页）

#### 封面页（固定格式）

| Shape | 内容 | 操作 |
|-------|------|------|
| [0] | 部门 + 日期 | 替换"2026年X月"为立项月份 |
| [1] | 公司名 + Charter文件 | **不动** |
| [2] | 项目名称 + 项目类别 | 替换项目名和类型 |

#### 目录页（固定）
- **不动**，保持模板原样。

#### 内容页（章节标题 + 子标题 + 自由内容区）

| Shape | 内容 | 操作 |
|-------|------|------|
| [0] 章节标题 | 如"一、项目背景" | **不动**（从目录来） |
| [1] 子标题 | 如"1.1 需求提升" | **只替换子标题文字** |
| 其余空白区域 | 正文内容 | **自由发挥**：文字/表格/图表/架构图均可 |

**子标题使用规则：**
- 一个章节**一页能讲完** → 不加子标题，直接放内容
- 一个章节**需要多页** → 加子标题拆分（如 3.1 技术价值 / 3.2 经济价值）
- **约束：内容不重叠、不超页面边界**

**内容形式不局限于文字：**
- 竞品对比 → 表格
- 行业分析 → 数据图表
- 技术架构 → 架构图
- 趋势分析 → 折线图/柱状图

#### 表格页（Q/C/D 指标、资源计划、财务、风险、结论评分）

| Shape | 内容 | 操作 |
|-------|------|------|
| [0] 章节标题 | 不动 | **不动** |
| [1] 小标题 | 如"Q指标" | **不动** |
| [2] 表格 | 数据表 | **清空数据行（保留表头）→ 填入大纲数据** |

**表格铁律：**
- 表格结构（表头、列数、样式）按模板，只填数据
- 模板中的示例数据仅作为样式参考
- 数据以大纲为准

#### 项目价值页（图表页）

| Shape | 内容 | 操作 |
|-------|------|------|
| [0] 章节标题 | 不动 | **不动** |
| 其余 | **图表/图形展示** | 不动或添加图表 |

- 一般没有子标题，尽量**一页用总分/并列/雷达图等图形展示**
- 多页是最后手段（加子标题拆分）
- 用 `<!-- chart: xxx -->` 标注图表类型

#### 产出和收益页（正文框）

| Shape | 内容 | 操作 |
|-------|------|------|
| [0] 章节标题 | 不动 | **不动** |
| [1] 正文框 | 收益内容 | **替换正文** |

#### 结论-总结页（混合固定+可变）

| Shape | 内容 | 操作 |
|-------|------|------|
| [0] 章节标题 | 不动 | **不动** |
| [1] "总结" | 不动 | **不动** |
| [2] 正文描述 | 项目值得做，做了能得到什么 | **替换** |
| [3] "建议项目经理为：{姓名}" | 用户姓名 | **替换** |
| [4] "恳请IPMT批准..." | 固定写法 | **不动** |

#### 固定不动的页面

| Slide | 页面 | 原因 |
|-------|------|------|
| 1 | 目录 | 固定模板格式 |
| 19 | 专业委员会评审意见 | 留给评审填写 |
| 20 | 结束页 | 固定模板格式 |

### 视觉增强：智能图表映射

大纲中的内容按以下规则映射为 PPT 智能图表，由 ppt-master 在生成时应用：

| 内容模式 | 识别特征 | 推荐图表类型 | 典型场景 |
|---------|---------|------------|---------|
| **总分结构** | 一个核心观点 + 多个支撑要点 | 中心辐射图 / 金字塔图 | 项目价值（技术+经济+应用+战略）、风险分类 |
| **对比分析** | 两组或多组对照数据 | 对比柱状图 / 雷达图 | 竞品对比、方案对比、计划 vs 实际 |
| **流程步骤** | 按顺序排列的阶段或步骤 | 流程图 / 时间轴 | 技术路线、研发阶段、里程碑 |
| **层级列表** | 有包含关系的分类信息 | 树状图 / 组织架构图 | 技术框架分层、团队架构、成果分类 |
| **数据表格** | 多维度数值数据 | 表格 + 条件着色 | Q/C/D 指标、费用明细、人力计划 |
| **趋势变化** | 随时间变化的数据 | 折线图 / 面积图 | 行业趋势、能效变化、成本下降 |
| **占比构成** | 整体与部分的关系 | 环形图 / 堆叠柱状图 | 费用构成、人力分配、成果分布 |

**标注方式：** 在大纲 md 文件中，用 HTML 注释标注期望的图表类型：

```markdown
## 三、项目价值
<!-- chart: radar -->
- 技术价值：建立凝霜识别核心算法能力
- 经济价值：预计降低化霜能耗 30%
- 应用价值：可复用到多产品线
- 战略价值：构建智能家电感知平台
```

ppt-master 读取大纲时识别 `<!-- chart: xxx -->` 标注，选择对应图表样式生成。

**常见标注参考：**
- `<!-- chart: radar -->` — 雷达图（多维度评估）
- `<!-- chart: timeline -->` — 时间轴（里程碑、路线图）
- `<!-- chart: pyramid -->` — 金字塔图（总分层次）
- `<!-- chart: comparison -->` — 对比图（两组数据对照）
- `<!-- chart: flow -->` — 流程图（步骤、决策）
- `<!-- chart: table -->` — 美化表格（数据展示）
- `<!-- chart: hierarchy -->` — 层级图（架构、分类）

### 生成方式

### 核心原则：母版不动，只改内容

| 铁律 | 说明 |
|------|------|
| **PPT 母版/模板不能变** | 页面布局、字体、颜色、公司名称、logo 位置、表格样式 — 全部保留 |
| **模板文字 = 样式标准** | 模板中已有的标题、副标题、表格内容的字体大小/粗体/颜色就是样式基准，替换内容时必须沿用，不能自由发挥 |
| **大纲改变正文内容** | 替换每页的要点文字（不是追加，是替换） |
| **大纲改变表格数据** | 清空表格旧数据 → 填入新数据（保留表头和格式） |
| **固定写法保留** | 结论"恳请IPMT批准..."、封面公司名称等 — 不改 |
| **内容形式自由** | 内容页可用文字/表格/图表/架构图，不局限于文字 |
| **不重叠不超界** | 所有内容不能重叠，不能超出页面边界 |

### 生成方式：两阶段协作

**阶段一：gen_ppt.py — 填充文字内容和表格**

```bash
python scripts/gen_ppt.py \
  --outline {project_base_dir}/{项目名}/大纲/{项目名}-{STAGE}-v{N}.md \
  --product-line hm \
  --output {project_base_dir}/{项目名}/PPT/
```

脚本处理：
- 封面页：替换项目名、类型、日期、**公司名和承担单位**（根据产品线自动替换）
- 内容页：Shape [1] 只放子标题，正文内容放在下方空白区域
- 表格页：清空数据行 → 填入大纲数据
- 结论页：替换总结内容和项目经理，保留固定写法
- 备注区：写入对应 slide 的备注

**阶段二：content_layout.py — 内容布局增强**

```bash
python scripts/content_layout.py \
  {PPT路径} \
  {大纲路径}
```

在模板 slide 上添加可视化卡片布局（不改背景）：
- 内容页（2-8）：三列卡片（半透明背景 + 色条 + 标题 + 要点）
- 项目价值页（9）：2×2 四维度卡片
- 产出收益页（15）：KPI 大数字卡片
- 原文移到备注区供讲解

**阶段二：/ppt-master — 生成图表页视觉内容**

gen_ppt.py 处理完文字和表格后，调用 ppt-master 生成图表页（如项目价值页）的视觉元素：

```
/ppt-master 为以下页面生成图表：
- Slide 9（项目价值）：总分图，展示技术/经济/应用/战略四维度价值
  内容来源：大纲中"三、项目价值"章节
  模板：charter-template.pptx 的 Slide 9
```

ppt-master 读取大纲中的 `<!-- chart: xxx -->` 标注，生成对应的 SVG 图表，嵌入到 PPT 的对应页面。

**绝不做的事：**
- ❌ 不改母版（页面布局、字体、颜色、公司名称）
- ❌ 不追加数据到表格（是清空后填入，不是 append）
- ❌ 不删除模板的固定元素
- ❌ 不添加模板中没有的页面
- ❌ 不用 ppt-master 重新生成整个 PPT（只用于图表页补充）

### 阶段五：检查审核清单

| 检查项 | 内容 | 通过标准 |
|--------|------|---------|
| **内容完整性** | 每页是否填入了大纲内容 | 无空白页（除图表页外） |
| **表格数据** | Q/C/D/资源/财务/风险/评分 | 数据正确，表头保留，样式一致 |
| **封面信息** | 项目名、类型、日期 | 全部替换，无 XXXXX 残留 |
| **固定写法** | 结论"恳请IPMT批准..." | 未被修改 |
| **溢出检查** | 所有 shape 边界 | 不超出页面 960×540 pt |
| **字体样式** | 正文、表格、标题 | 沿用模板字体，无自由发挥 |
| **备注区** | 各章节备注 | 已写入对应 slide |
| **逻辑连贯** | 章节顺序、内容衔接 | 符合 IPD 评审逻辑 |
| **价值论证** | 必要性+重要性+投入产出 | 有量化数据，逻辑清晰 |
| **产出收益** | 亮点数字 | 至少 3 项，有冲击力 |

## 工作流程

## 工作流程（五阶段）

### 阶段一：收集内容
- 用户描述项目（怎么说都行）
- 引导提问（要解决什么？带来什么？有没有资料？）
- 主动加工提炼（口语 → 评审语言，补充量化数据）
- 定向追问（缺什么补什么）
- 产出：**项目信息汇总**（口头或笔记）

### 阶段二：制定大纲
- 将收集的信息整理成结构化大纲 md
- 多轮迭代确认（v1 → v2 → v3...）
- 重点确认：价值论证、Q/C/D 指标、费用数据
- 产出：**确认版大纲** `{项目名}-{STAGE}-v{N}.md`

### 阶段三：填充 PPT
- 调用 gen_ppt.py，基于模板填充文字和表格
- 模板母版不动，只改内容
- 表格清空旧数据，填入大纲数据
- 产出：**内容完整的 PPT**（文字+表格已填充）

### 阶段四：美化渲染
- 调用 /ppt-master，生成图表页视觉内容
- 项目价值页 → 雷达图/总分图
- 产出和收益页 → KPI 大数字卡片
- 产出：**最终 PPT**（内容+图表完整）

### 阶段五：检查审核
- 内容完整性检查：每页内容是否填入，表格数据是否正确
- 视觉检查：溢出、字体、布局、样式是否正常
- 逻辑检查：价值论证是否充分，投入产出是否清晰
- 产出：**审核报告**（问题列表 + 修复建议）

## 大纲生成与确认

引导提问结束后，将收集到的所有信息整理成结构化大纲文件，经用户多轮确认后再生成 PPT。

### 大纲文件格式

保存到项目目录的 `大纲/` 子目录下，命名规则：`{项目名}-{STAGE}-v{N}.md`（N 为版本号，从 1 开始）。

完整路径：`{project_base_dir}/{项目名}/大纲/{项目名}-{STAGE}-v{N}.md`

文件结构：

```markdown
# {项目名} — {阶段名} 汇报材料

- 要点 2

<!-- chart: comparison -->
| 维度 | 我方 | 竞品A | 竞品B |
|------|------|-------|-------|
| ... | ... | ... | ... |

## 二、研究内容
<!-- chart: flow -->
### 2.1 技术路线
- 阶段一：数据采集 → 阶段二：模型训练 → 阶段三：部署验证
...

## 三、项目价值
<!-- chart: radar -->
- 技术价值：...
- 经济价值：...
- 应用价值：...
- 战略价值：...

## 备注区
## 一、项目背景
- 讲解要点：...
```

**正文区**：每章按 PPT 页面拆分，每页 3-5 个要点，层级清晰。
**备注区**：对应每章的讲解文案、数据支撑、过渡语。
**图表标注**：`<!-- chart: xxx -->` 标注在章节标题下方，指导 ppt-master 选择对应的智能图表样式。

### 表格页填写规则

模板中以下页面已有固定表格格式，大纲中只需提供数据行（表头可省略，脚本会自动跳过与模板重复的表头）：

| 页面 | 模板表头 | 大纲数据格式 |
|------|---------|------------|
| Q 指标 | 序号 / 指标要求 / 交付物 | `功能 \| 指标描述 \| 交付物` |
| C 指标 | 费用科目 / 预算 / 备注 | `费用项 \| 金额 \| 说明` |
| D 指标 | 关键节点（日期）/ 工作内容 / 交付物 | `2026.7.31 \| 工作内容 \| 交付物` |
| 关键资源计划 | 人员 / 角色 / 工作内容 / 计划工作量 | `姓名 \| 角色 \| 工作内容 \| 人天` |
| 财务计划 | 费用项目 / 预算金额 / 各阶段 | `费用项 \| 金额 \| 概念计划 \| 开发验证 \| 转移` |
| 风险管理 | 类别 / 风险名称 / 概率 / 影响 / 优先级 / 应对 / 责任人 | `性能 \| 风险描述 \| 中 \| 大 \| 高 \| 措施 \| 人` |
| 结论评分 | 评价指标 / 重要性 / 创新先进性 / 难度 | 模板已有示例，按实际评分填写 |
| ADCP 偏差 | 指标要求 / 实际达成 / 偏差 / 偏差分析 | `指标 \| 实际值 \| 偏差 \| 原因分析` |
| 转移完成 | 技术成果转移项 / 转移完成情况 / 备注 | `Q指标 \| 完成状态 \| 备注` |

**注意：**
- 表格数据行的列数应与模板表头列数一致
- 模板的固定写法（如结论"恳请IPMT批准..."）不要在大纲中出现，脚本会保留
- 每个阶段模板中已有示例数据，引导用户参考示例格式提供数据

### 多轮确认流程

1. **生成初版大纲（v1）**：将引导提问收集的信息整理成完整大纲，保存为 md 文件
2. **展示给用户**：输出大纲内容，逐章或逐页请用户审阅
3. **收集反馈**：用户指出需要修改、补充、删除的内容
4. **迭代更新**：修改后保存为新版本（v2、v3…），版本号递增
5. **最终确认**：用户明确表示"确认"/"可以了"/"没问题"后，锁定当前版本

**迭代原则：**
- 每轮迭代保存新版本，保留历史版本供回溯
- 重点确认：项目价值论证、Q/C/D 指标、费用数据、结论
- 如用户只修改个别章节，可局部更新后保存为新版本
- 确认完成后，该 md 文件即为 PPT 的内容蓝本

## 费用计算

- 人力成本基准：从 `config.json` 读取（默认 970 元/人天）
- 自动计算：人天数 × cost_per_day = 人力成本
- 其他费用需用户提供：服务器、差旅、试制、模具、专利等

## 注意事项

- 数据所项目核心关注：**投入产出、降本提效、业务/产品需求、行业技术布局**
- B级及以上项目需填写知识产权方案
- 项目编号、项目级别等信息需向用户确认
- 考核评分标准参考 `references/` 目录下的考核办法文档
